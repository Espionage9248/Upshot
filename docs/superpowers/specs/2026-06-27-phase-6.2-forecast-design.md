## 2. Why the V1 port is not a 1:1 copy (schema deltas)

Confirmed by reading the V2 schema (`packages/db/src/schema/`):

- **Scheduled outflows fan across three tables, not one.** V1 projected forecast
  events from a single `Subscription` table with kinds `BILL/SUBSCRIPTION/DEBT/
  AFTERPAY`. V2 splits these into `recurring_items` (kinds `BILL`/`SUBSCRIPTION`
  only), `debts` (`monthlyPaymentCents`), and `installment_plans` (BNPL —
  `installmentCents`, `totalInstallments`, `installmentsPaid`, `nextDueDate`,
  `frequencyDays`). The forecast must read all three.
- **Spending account = `role: "SPENDING"`** (V2 `ACCOUNT_ROLES`), not V1's
  `accountType: "TRANSACTIONAL"`.
- **No saver-funded-expense concept in V2.** V1 excluded bills "paid from a
  Saver" using `CategorySaverMapping` + an `isEssentials` Saver. V2 has **neither
  table nor flag** — it models savers via account `role`
  (`SAVER`/`ESSENTIALS`/`EMERGENCY`) + `monthlyAllocationCents` +
  `budget_allocations` (per-month). **Deviation:** drop the per-event
  saver-funded exclusion; model saver outflow as a single per-pay-cycle
  allocation deducted from the spending balance on each projected pay-day
  (V1's `perCycleAllocation`, derivable from `budget_allocations` / recent
  saver transfer inflows). Documented, intentional simplification.

Data flow: `page.tsx` (server) → `loadForecastData(db, { now, horizon })` → builds
the forecast DTO **for the requested horizon** (default 90; switched via the `?h`

`@upshot/core`; loaders/actions only marshal DB rows in and DTOs out.

## 4. `forecast.ts` — the engine (detail)

`buildCashflowForecast(input: ForecastInput, horizon: 30 | 60 | 90): CashflowForecast`

**Input (all integer cents / ISO strings; assembled by the web loader):**
- `nowISO`, `startBalanceCents` (the `role: SPENDING` account balance)
- `actualDailyNetCents: { dateISO; netCents }[]` — last ~30 days of spending-account
  net flow, for reconstructing the **actual** line by walking *backwards* from
  `startBalanceCents` (V2-new; the design spec wants an actual→projected join).
- `salary: { cadenceDays: 7|14|30; amountCents; lastPayISO } | null` — derived
  from recent `isSalary` txns (reuse the cadence heuristic already in
  `reports/salary-periods.ts`; fall back to `app_settings` monthly income).
- `scheduledOutflows: { dateISO; amountCents }[]` — fanned from `recurring_items`
  (active, `nextExpectedDate` + `frequency`), `debts` (`monthlyPaymentCents`,
  monthly), `installment_plans` (active, `nextDueDate` + `frequencyDays`, capped
  at `totalInstallments − installmentsPaid`).
- `perPayCycleAllocationCents` — saver outflow per pay-day (§2 deviation).
- `discretionary: { avgDailyCents; stdDevDailyCents }` — from 90-day
  non-transfer/non-salary spending-account outflows **excluding** amounts already
  represented as scheduled outflows (avoid double counting).
- `bandK` — confidence multiplier (default 1.0; `k·σ·√d`).

**Output `CashflowForecast`:**
```ts
interface ForecastDay { dateISO: string; centralCents: number;
                        lowCents: number; highCents: number }
interface CashflowForecast {
  startBalanceCents: number;
  actual: { dateISO: string; balanceCents: number }[];   // reconstructed history
  projected: ForecastDay[];                               // one per day [1..horizon]
  lowestProjectedCents: number; lowestDateISO: string;
  overdraftRisk: boolean;                                 // any central < 0
  horizon: 30 | 60 | 90;
}
```

**Algorithm:**
1. Reconstruct `actual[]`: from `startBalanceCents`, subtract each day's
   `netCents` backwards to get prior-day balances; reverse to chronological.
2. Project salary inflows across `[now, now+horizon]` from cadence; weekend
   pay-dates shift to the preceding Friday (Up pays on the last business day —
   port `toPrecedingWeekday`). Each pay-day also emits `perPayCycleAllocationCents`
   as an outflow.
3. Lay `scheduledOutflows` onto their dates.
4. Walk day 1..horizon: `central = prev + scheduledNet(day) − avgDailyCents`.
   Income applied before same-day outflows (prevents false overdraft).
5. Band at day d (1-indexed): `halfWidth = round(bandK · stdDevDailyCents · √d)`;
   `low = central − halfWidth`, `high = central + halfWidth`. Monotonic-widening.
6. Track lowest central + its date; `overdraftRisk = lowestProjectedCents < 0`.

Floats permitted only for `√d` and the band ratio; all returned cent values are
`Math.round`ed integers.

## 5. `salary-change.ts` / `expense-change.ts` (port)

Faithful ports of V1 `scenarioService.salaryChangeScenario` /
`expenseChangeScenario`, re-pointed at V2 tables, kept pure (DB reads happen in
the web loader; the core fn takes plain inputs):

- **Salary change** — `simulateSalaryChange({ currentMonthlyIncomeCents,
  newMonthlyIncomeCents, debts, savers, monthlyExpensesCents,
  monthlyExplicitSavingsCents })` → `{ incomeChangeCents, incomeChangePct,
  current/projectedSavingsRate, current/projectedDTI, additionalMonthlyFreedomCents,
  debtPayoffImpact | null, allocationSuggestions[] }`. Debt-payoff impact reuses
  the existing `core/src/payoff` snowball engine (baseline vs +incomeChange extra),
  **not** a re-implementation.
- **Expense change** — `simulateExpenseChange({ currentIncomeCents, savers,
  adjustments: { saverId; newAllocationCents }[], debts,
  monthlyExplicitSavingsCents })` → `{ adjustments[], current/newTotalAllocated,
  allocationChangeCents, current/projectedSavingsRate, monthlyImpactCents,
  yearlyImpactCents }`.

Savings-rate / DTI ratios are floats (percentages); all money is integer cents.

## 6. `forecast-chart.tsx` (UI)

Per design_handoff_upshot/"Upshot Component Specs.md" §"Cashflow / Forecast — SVG":
- **Actual** line = `--viz-1` (coral), solid.
- **Projected** tail = same hue, **dashed** (`stroke-dasharray`).
- **Confidence band** = filled area between `low`/`high`, coral at 12% via
  `color-mix(in oklch, var(--viz-1) 12%, transparent)` (NO hex).
- **"Today" divider** = vertical `--line` rule at the actual→projected join, labelled.
- **30/60/90** switch = `Tabs` (existing UI primitive); horizon changes reload via
  search param (`?h=30|60|90`), server-driven, matching the Reports page pattern.
- Overdraft: if `overdraftRisk`, a `--warn`-tinted callout naming `lowestDateISO`
  and the lowest balance (`Money kind="projected"`).
- Client component; serializable cents props only. Hand-SVG (matches the other
  finance charts; visx not needed for a line+band).
